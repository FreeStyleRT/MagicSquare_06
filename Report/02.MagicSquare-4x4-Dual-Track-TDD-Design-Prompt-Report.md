# Magic Square 4x4 - Dual-Track UI + Logic TDD 설계 프롬프트 보고서

작성일: 2026-05-28  
저장소: `MagicSquare_16`  
문서 목적: 설계 문서 생성 자체가 아니라, 설계 문서를 안정적으로 생성하기 위한 실행 프롬프트를 고정하고 운영 기준을 명확히 한다.

---

## 1. 배경 및 목적

본 작업의 핵심 목적은 알고리즘 구현보다 아래 3가지를 훈련하는 것이다.

1. 레이어 분리 (UI Boundary / Domain / Data)
2. 계약 기반 테스트 (Contract-first)
3. RED-GREEN-REFACTOR 사이클 하 리팩터링 규율

따라서 실행 프롬프트는 "구현 코드 생성"을 금지하고, 설계/계약/테스트/통합 계획만 출력하도록 강하게 제한한다.

---

## 2. 입력/출력 고정 계약

### 입력 계약

- 타입: `4x4 int[][]`
- `0`은 빈칸
- 빈칸 개수: 정확히 2개
- 값 범위: `0` 또는 `1~16`
- `0` 제외 중복 금지

### 출력 계약

- 타입: `int[6]`
- 좌표는 1-index
- 반환 형식: `[r1,c1,n1,r2,c2,n2]`
- `n1,n2`는 누락 숫자 2개
- `(작은수->첫빈칸, 큰수->둘째빈칸)` 배치가 마방진이면 그 순서 반환
- 아니면 반대 순서 반환

---

## 3. 프롬프트 설계 원칙

### 3.1 금지/강제 규칙

- 구현 코드, 의사코드, 알고리즘 절차 코드 금지
- 모호한 표현 금지 (`적절히`, `충분히`, `필요시`)
- 모든 규칙은 테스트로 검증 가능해야 함
- 출력 구조/제목/순서를 고정해 문서 편차를 줄임

### 3.2 레이어 관점

- Logic Layer: 불변조건, 유스케이스, 내부 계약, 단위 테스트 설계
- Screen Layer: 외부 계약, 에러 스키마, UI 계약 테스트 (Domain Mock 전제)
- Data Layer: 저장/로드 인터페이스, 구현 옵션 비교, 데이터 레이어 테스트
- Integration: 경로, 실패 시나리오, 회귀 보호, 커버리지 목표, 추적 행렬

---

## 4. 산출물 구조 표준

실행 프롬프트는 아래 구조를 강제한다.

1. Logic Layer (Domain)
2. Screen Layer (Boundary UI)
3. Data Layer
4. Integration & Verification

각 섹션은 체크리스트/표를 포함하고, 검증 가능한 Acceptance Criteria를 제시한다.

---

## 5. 품질 게이트

최종 산출물은 아래를 모두 만족해야 한다.

- 구현 코드 0줄
- 규칙-테스트 매핑 존재
- 레이어 책임 충돌 없음
- 입력/출력/에러 계약 고정
- Traceability Matrix 포함

커버리지 목표:

- Domain Logic 95%+
- UI Boundary 85%+
- Data 80%+

---

## 6. 운영 메모

- 본 문서는 "설계 문서 생성용 프롬프트"의 운영 기준 보고서다.
- 실제 설계 문서 작성은 `Prompting`에 저장된 실행 프롬프트를 별도 세션에서 실행해 생성한다.
- 브랜치 운영 시 `develop` 기준 `spec`부터 시작해 계약/테스트 설계를 고정한 뒤 단계별 진행하는 방식을 권장한다.

## 7. 부록: 사용 지시문(요약)

아래 조건으로 프롬프트를 실행한다.

- 역할: Dual-Track UI + Logic TDD 및 Clean Architecture 설계 전문가
- 목표: 설계/계약/테스트/통합 계획 문서 산출
- 제약: 구현 코드 금지
