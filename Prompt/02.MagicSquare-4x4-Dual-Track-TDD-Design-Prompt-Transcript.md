## Final Executable Prompt (원문 보관)

```text
당신은 **Dual-Track UI + Logic TDD** 및 **Clean Architecture 설계 전문가**입니다.

프로젝트: **Magic Square (4x4) — TDD 연습용**  
핵심 목적: 알고리즘 난이도보다 **레이어 분리 + 계약 기반 테스트 + 리팩토링 규율** 훈련

---

## 전역 제약 (반드시 준수)

1. **구현 코드 작성 금지** (함수 본문, 알고리즘 코드, 의사코드 포함 금지)
2. 설계/계약/테스트/통합 계획만 작성
3. UI는 실제 화면이 아니라 **입력/출력 Boundary**로 정의
4. Data Layer는 DB가 아니라 **저장/로드 인터페이스** 수준만 다룸 (메모리/파일 교체 가능)
5. 모호한 표현 금지: `적절히`, `충분히`, `필요시` 사용 금지
6. 모든 규칙은 **테스트로 검증 가능**해야 함

---

## 입력/출력 계약 (고정)

### 입력 계약
- 타입: `4x4 int[][]`
- `0`은 빈칸
- 빈칸(`0`)은 **정확히 2개**
- 값 범위: `0` 또는 `1..16`
- `0` 제외 중복 금지

### 출력 계약
- 타입: `int[6]`
- 좌표는 **1-index**
- 반환 형식: `[r1,c1,n1,r2,c2,n2]`
- `n1,n2`는 두 누락 숫자
- `(작은수→첫빈칸, 큰수→둘째빈칸)` 배치가 마방진이면 그 순서 반환
- 위 조합이 실패하면 반대 순서로 반환

## 출력 형식 (아래 구조를 제목/순서 그대로 유지)

# 1) Logic Layer (Domain Layer) 설계
## 1.1 도메인 개념
- Entities / Value Objects / Domain Services 목록
- 각 항목의 단일 책임(SRP)

## 1.2 도메인 불변조건(Invariants)
- 행/열/대각선 합 일치
- Magic Constant
- 입력 제약 관련 불변조건 포함

## 1.3 핵심 유스케이스(도메인 관점)
- 빈칸 찾기
- 누락 숫자 찾기
- 마방진 판정
- 두 조합 시도 및 선택 규칙

## 1.4 Domain API(내부 계약)
- 메서드 시그니처 수준(코드 금지)
- 입력/출력/실패조건 명시

## 1.5 Domain 단위 테스트 설계(RED 우선)
- 정상/비정상/엣지 테스트 목록
- 각 테스트가 보호하는 invariant를 명시

---

# 2) Screen Layer (UI Layer) 설계 (Boundary Layer)
## 2.1 사용자/호출자 관점 시나리오
- “행렬 입력 → 검증 → 결과 출력” 흐름

## 2.2 UI 계약(외부 계약)
- Input schema
- Output schema
- Error schema

## 2.3 UI 레벨 테스트(Contract-first, RED 우선)
- 잘못된 크기
- 빈칸 개수 오류
- 값 범위 오류
- 중복 오류
- 반환 포맷 검증
- Domain은 Mock으로 가정

## 2.4 UX/출력 규칙
- 에러 메시지 표준
- 메시지 문구 규칙(일관된 형식) 명시

---

# 3) Data Layer 설계 (Data Layer)
## 3.1 목적 정의
- 저장/로드 필요성
- 학습용 범위 한정

## 3.2 인터페이스 계약
- 예: `MatrixRepository.save/load` (메서드 수준, 코드 금지)
- 저장 대상: 입력 행렬, 실행 결과(선택)

## 3.3 구현 옵션 비교(메모리/파일)
- 옵션 A: InMemory
- 옵션 B: File(JSON/CSV)
- 추천안 1개 선택 + 선택 근거

## 3.4 Data 레이어 테스트
- 저장/로드 정합성
- 예외: 파일 없음/형식 오류
- 불변조건: 4x4 유지

---

# 4) Integration & Verification (통합 및 검증)
## 4.1 통합 경로 정의
- UI → Application(선택) → Domain → Data
- 의존성 방향과 경계 위반 금지 규칙

## 4.2 통합 테스트 시나리오
- 정상 시나리오 2개 이상
- 실패 시나리오 3개 이상:
  - 입력 오류
  - 도메인 실패
  - 데이터 실패

## 4.3 회귀 보호 규칙
- 기존 테스트 유지 정책
- 변경 금지 규칙: 계약/출력 포맷

## 4.4 커버리지 목표
- Domain Logic 95%+
- UI Boundary 85%+
- Data 80%+

## 4.5 Traceability Matrix (필수)
- 열 순서 고정:
  `Concept(Invariant) | Rule | Use Case | Contract | Test | Component`
- 최소 10행 이상 작성
```
